# java-pattern-instructor

根据需求场景，指导如何正确选择和落地 GoF 23 种设计模式。

**核心职责**：
- 帮助识别需求中的变化点与稳定点
- 指导选择最匹配的设计模式
- 给出最小必要实现的代码结构

**输出结构**：分析需求 → 给出模式选择理由 → 提供实现骨架 → 说明扩展方向和避坑

---

## 与 java-pattern-reviewer 的区别

| skill | 职责 |
|-------|------|
| java-pattern-instructor（本 skill） | 从零开始实现设计模式，给出正确落地方式 |
| java-pattern-reviewer | 审查已有设计模式实现是否合理 |

**使用边界**：如果已有实现但不确定是否合理，用 java-pattern-reviewer；如果还不知道怎么落地，用 java-pattern-instructor。

---

## 适用场景

1. **需求分析阶段**：拿到新需求，判断是否需要设计模式、上哪种模式
2. **技术方案设计**：确认模式落点、层次划分、接口定义
3. **代码实现前**：确认实现思路，参考模式标准实现骨架
4. **学习验证**：验证对某个设计模式的理解是否正确

## 核心指导原则

### 1. 先识别再选择

**先确认需求中的三要素**：
- 稳定主干：什么是不变的
- 变化点：什么会扩展或变化
- 协作边界：谁会负责维护哪部分

**再匹配模式**：
- 变化点匹配不到具体模式时，优先轻量封装
- 变化点明确但模式不匹配时，不要强行模式化
- 只有 1-2 个变化分支时，优先 if-else 或简单注入

### 2. 模式选择判断树

```
是否有多种可替换算法/策略？
  └─ YES → 策略模式 / 状态模式（看是否带状态迁移）
  └─ NO

主流程稳定但步骤有差异？
  └─ YES → 模板方法模式
  └─ NO

对象创建复杂或需要解耦创建者？
  └─ YES → 工厂模式 / 建造者模式
  └─ NO

需要把类接口转换成客户端期望的接口？
  └─ YES → 适配器模式
  └─ NO

需要让对象在内部状态变化时自动行为？
  └─ YES → 状态模式
  └─ NO

需要定义对象间一对多依赖关系？
  └─ YES → 观察者模式
  └─ NO

需要递归组合树形结构？
  └─ YES → 组合模式
  └─ NO

需要用对象封装替代类型码/分支？
  └─ YES → 桥接模式
  └─ NO

其他情况 → 优先不要上模式，先轻量实现
```

### 3. 抽象粒度原则

- 接口数量刚好：够用但不过度抽象
- 类职责单一：每个类只做一件事
- 避免空接口、空父类、过度工程化
- 优先组合，轻量依赖注入优于继承树

### 4. 简单优先原则

| 替代方案 | 适用场景 |
|----------|----------|
| if-else | 变化分支 ≤ 2 |
| 简单注入 | 只需要一个实现 |
| 枚举 + 策略 | 状态转换简单 |

## GoF 23 种模式快速参考

### 创建型（5种）

| 模式 | 适用场景 | 核心实现 |
|------|----------|----------|
| Singleton | 全局唯一配置/资源访问器 | 枚举单例最安全 |
| Factory Method | 创建逻辑可能扩展，下沉到子类 | 产品接口 + 创建者抽象 + 具体创建者 |
| Abstract Factory | 创建一族相关对象 | 产品族接口 + 具体工厂 |
| Builder | 对象创建需多步骤/大量可选参数 | 私有构造 + 静态内部 Builder |
| Prototype | 对象创建成本高，需要复制 | Cloneable + 深拷贝处理 |

### 结构型（7种）

| 模式 | 适用场景 | 核心实现 |
|------|----------|----------|
| Adapter | 集成第三方/统一多个不同接口 | 目标接口 + 待适配类 + 适配器（组合优先） |
| Decorator | 动态添加职责，运行时组合 | 组件接口 + 装饰器基类 + 具体装饰器 |
| Proxy | 远程代理/懒加载/权限控制 | 与真实主题实现同一接口，内部持有真实主题引用 |
| Bridge | 两个独立维度变化 | 实现接口 + 抽象类组合实现 |
| Composite | 树形结构，部分-整体 | 组件接口 + 叶子节点 + 容器节点 |
| Facade | 为复杂子系统提供统一入口 | 门面类封装子系统调用 |
| Flyweight | 大量相同对象可共享 | 享元工厂 + 内部状态/外部状态分离 |

### 行为型（11种）

| 模式 | 适用场景 | 核心实现 |
|------|----------|----------|
| Strategy | 多种算法可替换，运行时选择 | 策略接口 + 具体策略 + 上下文 |
| Template Method | 主流程稳定，部分步骤可变 | 抽象模板类 + 具体子类覆写差异步骤 |
| Chain of Responsibility | 多个对象形成链处理请求 | 处理器接口 + 抽象基类 + 具体处理器 |
| State | 状态驱动行为变化，状态转换复杂 | 状态接口 + 具体状态 + 上下文 |
| Observer | 一对多依赖，状态变化通知 | 主题接口 + 观察者接口 + 具体主题 |
| Command | 请求封装，支持撤销/日志/队列 | 命令接口 + 具体命令 + 调用者 |
| Iterator | 遍历聚合对象不暴露内部 | 迭代器接口 + 聚合接口 |
| Mediator | 对象间通信复杂，集中管理 | 中介者接口 + 同事类 |
| Memento | 状态快照，撤销/回滚 | 原发器 + 备忘录 + 负责人 |
| Interpreter | 特定领域语言/规则引擎 | 表达式接口 + 终结符 + 非终结符 |
| Visitor | 对象结构稳定，操作经常变 | 元素接口 + 访问者接口 + 具体访问者 |

## 硬约束

1. **先识别再推荐**：必须先分析需求结构，不能不问需求就直接推荐模式
2. **简单优先**：能用 if-else 解决的不用策略，能用注入解决的不用工厂
3. **渐进式设计**：不要一开始就设计完整模式，先轻量落地，根据需要再演进
4. **不强推模式**：如果当前需求不需要模式，要明确告知不要过度设计
5. **上下文敏感**：考虑 Java 版本特性和并发场景
