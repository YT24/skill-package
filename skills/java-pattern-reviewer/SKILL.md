# java-pattern-reviewer

审查 Java 设计模式实现是否合理，输出整改建议型报告。

**核心原则**：先指出问题 → 再给最小整改方案 → 避免整套推翻重构

**不做什么**：不讲设计模式概念，不直接替代实现设计

---

## 适用场景

1. **开发中纠偏**：已引入策略/责任链/状态/适配器等结构，怀疑是否抽象过度
2. **提交前审查**：准备提交前，确认设计是否真的降低了复杂度
3. **重构审查**：判断哪些设计应保留，哪些应收敛，哪些应回退到轻量实现

---

## 审查输入

**自然语言**：需求目标、使用的模式、当前痛点

**代码上下文**（可选）：类名、包路径、关键方法、代码片段

> 上下文不足时，必须先提问澄清，不能直接做强结论

---

## 审查决策树

### 1. 模式适配性 → 当前模式是否真的适合当前需求？
- 分支很少却硬上策略 → 过度设计
- 流程不稳定却硬上模板方法 → 回退轻量
- 节点强依赖却硬上责任链 → 伪解耦
- 状态只是展示值却硬上状态模式 → 不应引入

### 2. 抽象粒度 → 抽象是否刚好？
- 空工厂、空父类 → 删除空抽象
- 一层接口包一层接口 → 过度封装
- 只有 2 个实现却拆很多类 → 收益不足
- 过早引入通用上下文对象 → 过度通用

### 3. 扩展性 → 新增是否以新增实现为主？
- 新增一个规则/状态/节点仍要改主流程 → 未获得开闭收益
- 入口仍保留大量 if-else → 策略模式未真正落地

### 4. 协作边界 → 接口是否稳定可独立维护？
- 多人仍频繁改同一个类 → 伪解耦
- 实现类之间强耦合 → 边界不清

### 5. 层次落点 → 类/接口是否放在合理位置？
- 模式逻辑是否在正确的包/层次
- 是否存在跨层调用混乱

## GoF 模式专项审查要点

### 创建型（Creational）

| 模式 | 重点审查 |
|------|----------|
| Singleton | 真的只需要一个实例？是否导致隐含全局状态 |
| Factory Method | 工厂是否只是包一层空调用 |
| Abstract Factory | 产品族是否真的稳定，引入成本是否过高 |
| Builder | 构建步骤是否真的复杂到需要建造者 |

### 结构型（Structural）

| 模式 | 重点审查 |
|------|----------|
| Adapter | 适配器是否只负责兼容转换，业务层仍充满被适配对象的字段判断 → 未真正隔离 |
| Decorator | 装饰器是否真的需要动态组合，继承是否足够 |
| Facade | 门面是否只是包一层空调用，底层是否真的复杂 |
| Proxy | 代理是否真的需要远程/懒加载，是否过度封装 |

### 行为型（Behavioral）

| 模式 | 重点审查 |
|------|----------|
| Strategy | 入口是否仍保留大量 type 判断，策略类是否极薄 → 应合并 |
| Template Method | 子类大量覆写，父类骨架无约束力 → 应用组合替代继承 |
| Chain of Responsibility | Handler 之间互相依赖内部状态 → 伪解耦 |
| State | 状态只是展示值，引入后主流程仍保留大段 status 分支 → 状态迁移未集中 |
| Command | 命令是否真的需要支持撤销/排队，调用者和接收者是否真的需要解耦 |
| Observer | 观察者是否真的需要动态注册，发布者和订阅者是否真的需要松耦合 |
| Iterator | 迭代器是否真的需要内部迭代，是否可以用 stream 替代 |
| Visitor | 元素结构是否真的稳定，双重分派是否真的必要 |

## 最小整改优先级

| 动作 | 适用场景 |
|------|----------|
| 合并类 | 弱价值策略类、相似实现 |
| 删除空抽象 | 空工厂、空父类、无收益接口 |
| 上移/下沉逻辑 | 错位的职责 |
| 回退轻量实现 | 模式明显不适合当前需求 |

---

## 硬约束

1. **允许建议删除**：模式明显不适合时，明确指出应退回轻量实现
2. **不用模式默认合理**：基于变化点识别、抽象层级、扩展收益判断
3. **优先最小整改**：能合并就不重构，能收敛就不推翻
4. **收益必须可验证**：落到"新增需求是否少改旧代码"
5. **上下文不足先提问**：缺少关键信息时不能做强结论
