# Overview

## java-pattern-reviewer

这个 skill 用于审查 Java / Spring Boot / Spring Cloud / MyBatis-Plus 项目中设计模式的落地是否合理。

它不是用来讲设计模式概念，也不是直接替代实现设计，而是重点回答下面这些问题：

- 当前模式是否真的匹配需求中的变化点
- 当前抽象是否过重、过早、过多
- 模式是否落错层
- 当前实现是否真正让主流程更稳定
- 当前扩展是否主要通过新增实现完成，而不是频繁修改旧逻辑
- 当前多人协作边界是否更清晰

这个 skill 的输出偏整改建议型：

- 先指出问题
- 再给最小整改方案
- 避免动不动建议整套推翻重构

---

# 适用场景

在以下场景优先使用这个 skill：

## 1. 开发中纠偏

当你在开发过程中已经引入了策略模式、责任链、状态模式、适配器等结构，但开始怀疑是否抽象过度、类拆分失衡、模式落错层时，使用这个 skill 做中途审查。

## 2. 提交前审查

当你准备提交代码、合并分支或发起评审前，希望确认当前设计模式实现是否真的降低了复杂度、是否会给后续维护带来问题时，使用这个 skill 做结构性审查。

## 3. 重构审查

当你面对已有复杂业务代码，已经做过一轮模式化重构，或者正在考虑删除部分抽象时，使用这个 skill 判断哪些设计应保留，哪些应收敛，哪些应回退到轻量实现。

# 输入方式

这个 skill 支持混合型输入。

你可以只提供自然语言描述，例如：

- 当前需求目标
- 当前使用的设计模式
- 当前结构和痛点
- 为什么怀疑设计不合理

也可以补充更精确的代码上下文，例如：

- 类名
- 包路径
- 关键方法
- 关键代码片段
- 主要调用链

如果上下文不足，必须先提出澄清问题，不能直接给强结论。

# 项目上下文与审查边界

该 skill 默认面向 Spring Boot 微服务项目，且遵循以下常见分层边界：

## 1. Controller

- 负责接参、参数校验、调用业务入口、返回结果
- 不承载复杂流程编排
- 不承载策略选择、责任链装配、状态迁移等模式主逻辑

## 2. Business

- 负责复杂业务编排
- 适合作为模式主入口
- 适合承载策略分发、责任链装配、状态流转入口、模板流程入口、事件发布入口

## 3. Service

- 提供原子业务能力
- 保持单一职责
- 不应演化成大而全的业务中枢

## 4. Mapper

- 只负责数据访问
- 不承载模式逻辑和业务判断

## 5. DTO / VO

- DTO 放在 `api/dto`
- VO 放在 `api/vo`
- DTO 以 `DTO` 结尾
- VO 以 `VO` 结尾
- 不使用内部类 DTO / VO

## 6. MyBatis-Plus 审查基线

- 查询优先使用 `lambdaQuery()`
- 避免硬编码字段名
- 默认无需重复追加 `delete_flag=0`
- 模式逻辑不应下沉到 Mapper 层

# 核心审查目标

每次审查都必须围绕以下问题展开：

1. 当前模式是否真的匹配需求中的变化点
2. 当前实现是否保护住了稳定主干
3. 当前抽象层级是否刚好，而不是过重或不足
4. 新增需求时，是否以新增实现为主，而不是反复修改主流程
5. 当前协作边界是否更清晰，而不是让多人继续修改同一个超长类
6. 当前实现是否引入了新的耦合、调试成本和维护负担
7. 当前模式是否真正解决了问题，而不是只增加了抽象感

---

# 通用审查维度

## 1. 模式适配性审查

重点判断：

- 当前模式是否真的适合当前需求
- 是否本来只需要简单封装，却提前做了完整模式化设计
- 模式是否匹配当前变化点

典型信号：

- 分支很少却硬上策略模式
- 流程并不稳定却硬上模板方法
- 节点强依赖却硬上责任链
- 状态只是展示值却硬上状态模式

## 2. 层次落点审查

重点判断：

- Controller 是否越界承担策略选择或流程编排
- Business 是否承担了应承担的主入口职责
- Service 是否变成了大而全的业务中枢
- Mapper 是否混入业务逻辑
- 相关类是否放在合理包结构中

## 3. 抽象粒度审查

重点判断：

- 抽象是否刚好
- 是否为了通用而过度通用
- 是否存在一层接口包一层接口
- 是否类数量明显膨胀但收益不足

常见问题信号：

- 空工厂
- 空父类
- 过早引入通用上下文对象
- 明明只有 2 个实现，却拆出很多类

## 4. 扩展性审查

重点判断：

- 新增一个规则、一个状态、一个节点、一个厂商时，是否主要通过新增实现完成
- 主流程是否仍需要频繁修改
- 当前实现是否真正获得了开闭原则的收益

## 5. 协作边界审查

重点判断：

- 接口是否稳定
- 实现类是否可以独立维护
- 是否减少了多人频繁改同一个 Business / Service
- 是否存在“看起来分开了，实际上仍强耦合”的伪解耦

## 6. Spring 落地合理性审查

重点判断：

- 是否适合用 Spring Bean 管理
- 是否合理使用 `Map<String, Strategy>`、`List<Handler>` 等装配方式
- 是否正确使用 AOP、事务、分布式锁、事件机制
- 是否存在事务边界、异步一致性、代理失效等问题

## 7. 渐进式改造合理性审查

重点判断：

- 当前改造是否一步过重
- 是否可以先轻量落地
- 是否为了理想结构一次性重构过大
- 是否提高了回归风险

# 常见模式专项审查规则

## 1. 策略模式 Strategy

重点审查：

- 是否真的存在多规则可替换
- 策略接口是否稳定
- 策略之间是否互相调用
- 入口中是否仍然保留大量类型判断
- 是否本来只靠 Spring 注入就够了，却额外叠加空工厂

常见问题信号：

- 策略类很多，但每个实现逻辑极薄
- Business 中仍有大量 `if-else`
- 新增一个策略仍要改主流程很多地方

## 2. 模板方法模式 Template Method

重点审查：

- 主流程是否真的稳定
- 差异步骤是否清晰
- 是否出现过深继承
- 是否本应用组合却用了继承

常见问题信号：

- 子类大量覆写，父类骨架没有实际约束力
- 不同流程差异太大，硬套同一模板

## 3. 责任链模式 Chain of Responsibility

重点审查：

- 节点是否真正独立
- 节点是否支持清晰排序和增删
- 节点之间是否强依赖
- 是否只是把一个超长方法拆成多个强耦合 Handler

常见问题信号：

- Handler 之间互相依赖内部状态
- 链路看起来可扩展，实际改动仍集中在入口和上下文对象

## 4. 状态模式 State

重点审查：

- 状态是否真的驱动行为变化
- 状态迁移规则是否集中
- 主入口是否仍充满状态判断
- 状态类边界是否清晰

常见问题信号：

- 状态只是展示字段
- 状态类之间耦合严重
- 引入状态类后主流程依然保留大段 `status` 分支

## 5. 工厂模式 Factory

重点审查：

- 工厂是否只负责选择实现
- 是否把复杂业务编排塞进工厂
- 是否只是包一层空调用
- 是否本来 Spring 注入足够，却多加一层没有价值的工厂

## 6. 适配器模式 Adapter

重点审查：

- 是否真正隔离了第三方差异
- 适配器是否只负责兼容和转换
- 业务层是否仍然充满第三方字段判断
- Adapter 是否反向依赖大量业务决策

## 7. 观察者 / 事件模式

重点审查：

- 后置动作是否真正与主流程解耦
- 是否处理好了事务边界和异步一致性
- 是否把强一致主流程误拆成异步监听
- Listener / Consumer 是否反向污染主流程

## 8. AOP / 代理模式

重点审查：

- 横切逻辑是否真的适合 AOP
- 是否把领域规则误抽成切面
- 事务、权限、日志、锁边界是否合理
- 是否存在自调用导致事务失效等问题

# 输出格式要求

每次审查时，按以下结构输出：

## 一、审查结论

- 当前方案总体是否合理
- 当前问题等级：高 / 中 / 低
- 是否存在过度设计
- 是否建议继续沿用当前方案

## 二、需求与模式匹配度

- 当前需求目标
- 当前变化点
- 当前稳定主干
- 当前模式是否匹配变化点
- 不匹配的原因（如果有）

## 三、结构问题审查

- Controller 是否越界
- Business 是否承担正确职责
- Service 是否过胖 / 过散
- Mapper 是否越界
- 包结构是否合理
- DTO / VO 是否符合边界

## 四、模式实现问题审查

- 当前模式实现的优点
- 当前模式实现的缺陷
- 是否存在抽象过度
- 是否存在抽象不足
- 是否存在未来扩展仍要修改主流程的问题

## 五、协作与扩展性审查

- 接口是否稳定
- 实现类是否可独立维护
- 是否减少了多人冲突
- 新增需求时主要是新增实现还是修改旧类
- 是否达到了预期扩展收益

## 六、Spring 落地审查

- Spring Bean 使用是否合理
- 事务 / AOP / 锁 / 事件机制是否合理
- 是否存在事务边界问题
- 是否存在异步一致性风险
- 是否存在远程调用适配层污染问题

## 七、整改建议

必须给出：

- 应保留的设计
- 应删除的抽象
- 应合并的类
- 应拆分的职责
- 应上移 / 下沉的逻辑
- 是否建议退回轻量实现

## 八、最小整改方案

只给最小必要调整建议，说明：

- 第一步改哪里
- 第二步收敛哪里
- 如何降低回归风险

## 九、风险提醒

- 如果继续维持当前方案，最大的维护风险是什么
- 如果立即整改，最大的改造风险是什么
- 哪些点必须补测试

# 使用示例

## 示例 1：开发中纠偏

输入：

```text
当前在人员标签计算模块里用了策略模式。
现在有 5 个策略类，但每个策略类只有少量逻辑，Business 里仍然保留大量 type 判断。
想确认是不是抽象过度。
```

预期审查重点：

- 策略模式是否真正落地
- 策略接口是否稳定
- Business 是否仍然承担过多分支判断
- 是否应先合并弱价值策略类，再收敛入口分发逻辑

## 示例 2：提交前审查

输入：

```text
审批流模块使用了责任链模式。
当前有多个 Handler，通过 Spring 注入装配，但部分 Handler 会直接依赖前一个 Handler 的内部结果对象。
准备提交前，想确认这种设计是否合理。
```

预期审查重点：

- Handler 是否真正独立
- 当前责任链是否只是拆开的超长方法
- 是否需要收敛链路上下文
- 是否需要把节点共享逻辑收回 Business 或独立 Service

## 示例 3：重构审查

输入：

```text
对接第三方平台时新增了 Adapter 层，但 Service 中仍然保留大量第三方返回字段判断。
当前涉及 Feign 调用和事务处理，想判断 Adapter 是否真正起作用。
```

预期审查重点：

- 第三方差异是否真正被隔离
- Adapter 是否承担了正确职责
- Service 是否仍被外部字段污染
- 是否需要先统一内部模型，再收敛适配层边界

# 硬约束

## 1. 允许明确建议删除当前模式

如果当前设计模式明显不适合，必须明确指出：

- 应删除当前模式实现
- 应退回简单实现或轻量封装
- 原因是什么

## 2. 不因为用了模式就默认合理

必须基于以下维度判断，而不是基于“代码看起来很抽象”：

- 变化点是否识别准确
- 稳定主干是否保护住了
- 抽象层级是否刚好
- 扩展是否以新增实现为主
- 协作边界是否更清晰

## 3. 优先最小整改

如果问题可以通过以下方式解决，就不要直接建议大重构：

- 合并类
- 删除空抽象
- 收回职责
- 上移 / 下沉少量逻辑
- 回退到轻量实现

## 4. 模式收益必须可验证

审查结论必须落到这些可验证问题上：

- 新增需求是否少改旧代码
- 主流程是否更清晰
- 多人协作是否更少冲突
- 排查问题是否更容易

## 5. 上下文不足时必须先提问

如果缺少需求背景、模式落点、主要类结构、当前痛点等关键信息，必须先提问澄清，不能直接做强结论。
